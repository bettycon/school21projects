## Задание 7. Разработать следующий пайплайн:

build — сборка приложения (запускать автоматически для веток с префиксом feature_);

test — запуск модульных тестов и функциональных тестов postman через утилиту newman (запускать автоматически для веток с префиксом feature_);

staging — запуск приложения в staging-окружении (запускать мануально и только для тегов).

Создаём файл с пайплайном .gitlab-ci.yml, в котором последовательно описываем требуемые стадии:

- Стадия build здесь собираем докер образы сервисов и пушим их в докер-хаб репозиторий. Тег образа - SHA коммита. Build запускается автоматически при каждом пуше в ветку с префиксом feature_.

![](../misc/images/23.png) </br>

![](../misc/images/24.png) </br>

![](../misc/images/25.png) </br>

 - Стадия test. Запускается автоматически для веток с префиксом feature_ посе успешного завершения стадии build. Создаётся временный неймспейс. Выполняем развертывание Helm-релиза в в Kubernetes-кластере, используя Docker-образы, с тегом SHA коммита из предыдущей джобы, без Ingress, и ждёт полной готовности всех компонентов. После чего устанавливаем и запускаем утилиту newman, при помощи имеющейся коллекции тестов проверяем доступность эндпоинтов микросервисов. Удаляем временный неймспейс.  

![](../misc/images/28.png) </br>

![](../misc/images/26.png) </br>

![](../misc/images/27.png) </br>

- Стадия staging. Запускается вручную и только для тегов. Разворачиваем микросервисы в созданном неймспейсе. Используем Docker-образы, с тегом SHA коммита из первого джоба, запускаем Ingress, оставляем созданный неймспейс.

![](../misc/images/29.png) </br>

Создаем новый тег, пушим в ветку гитлаба и запускаем вручную стадию staging

![](../misc/images/30.png) </br>
